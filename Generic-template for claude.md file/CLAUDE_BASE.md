# CLAUDE_BASE.md — Generic Base Rules

> **Do not modify this file.** It contains universal rules that apply to every project.
> Project-specific instructions live in `CLAUDE.md` (see the Two-Layer System section below).

---

## How This System Works

This project uses a two-layer instruction system:

**Layer 1 — `CLAUDE_BASE.md` (this file)**
Universal rules that apply to every project, regardless of tech stack or domain. Maintained once, reused everywhere. Never modified per-project.

**Layer 2 — `CLAUDE.md` (project-specific)**
Generated by Claude at project start from a `PROJECT_BRIEF.md` written by the developer. Evolves as the project grows — Claude updates it when the tech stack is confirmed, architecture shifts, or major decisions are made. Always read this file at the start of every session.

**Layer 3 — `PROJECT_BRIEF.md`**
A short human-authored document describing what the project is, its domain, and any constraints or decisions already made. Claude reads this once during project setup and folds it into `CLAUDE.md`.

**Rule:** `CLAUDE_BASE.md` takes precedence over `CLAUDE.md` in all conflicts. Base rules cannot be softened or overridden by project-specific rules.

## Session Start Checklist

At the start of every session, before doing anything else:

1. **Bootstrap check** — verify the following files exist. If any are missing, create them immediately using the scaffolding in `templates/` before proceeding. Do not skip this step and do not proceed with any task until all files are present.

   | Required file | Template |
   |---------------|----------|
   | `CLAUDE.md` | `templates/CLAUDE.md` |
   | `DECISIONS.md` | `templates/DECISIONS.md` |
   | `ARCHITECTURE.md` | `templates/ARCHITECTURE.md` |
   | `TECH_DEBT.md` | `templates/TECH_DEBT.md` |
   | `ONBOARDING.md` | `templates/ONBOARDING.md` |

   If `PROJECT_BRIEF.md` exists but `CLAUDE.md` does not, generate `CLAUDE.md` from the brief before continuing.

2. Read `CLAUDE.md` (project-specific rules and current state)
3. Read `DECISIONS.md` (what decisions have been made and why)
4. Read `ARCHITECTURE.md` (current structure of the system)
5. Confirm the task with the user before beginning

## Behavioral Rules

### 1. State Assumptions Before Coding
Before writing any code, explicitly state what you are assuming about requirements, existing behavior, data shapes, and constraints. If an assumption is wrong, it should be caught before code is written, not after.

### 2. Read Before Write
Before editing any file, read it in full. No edits based on memory of what the file "probably" contains. No partial reads. If a file is too large to read fully, read the relevant sections and state which sections were skipped.

### 3. Simplicity First
Always prefer the simplest solution that satisfies the requirement. Do not add abstractions, generalization, or future-proofing unless explicitly asked. If you think a more complex solution will be needed later, flag it — do not build it now.

### 4. Surgical Changes Only
Only modify what is necessary to complete the current task. Do not refactor, clean up, or improve unrelated code while working on something else. If you notice something that should be fixed, log it to `TECH_DEBT.md` and move on.

### 5. Scope Discipline
Complete the assigned task fully before touching anything adjacent. Do not start a related improvement mid-task. If scope needs to expand, stop, surface it to the user, and get confirmation before proceeding.

### 6. Don't Invent Requirements
When something is ambiguous, stop and ask. State specifically what is unclear. Do not make a guess and bury it in a comment or assume the user will catch it in review. Unasked-for assumptions are bugs waiting to happen.

### 7. Fail Loudly
If something goes wrong, report it immediately and explicitly. Do not silently work around a failure, swallow an error, or continue as if it did not happen. A visible failure is always better than a hidden one.

### 8. Checkpoint Long Tasks
For any task that involves more than 3 distinct steps or touches more than 3 files, pause after each logical chunk and confirm before proceeding. Do not run to completion on a long task without checkpoints.

## Verification Rules

### 9. Verification Gate
After completing any task, run the relevant verification command (tests, lint, build, type check) before reporting done. Paste the command and its output. Never self-report success — let the output speak.

If no verification command exists for the task, say so explicitly and flag it as a gap.

### 10. Rollback Awareness
If a change breaks something:
- Default to `git revert` first, not fix-forward
- Fix-forward only if reverting is not possible or the user explicitly asks
- Never delete or overwrite uncommitted work without confirmation

## Tracking File Rules

These four files must be kept current. Update them as a side effect of normal work — not as a separate documentation pass.

| File | Mode | Update when |
|------|------|-------------|
| `DECISIONS.md` | Append-only | A non-obvious choice was made — tech, architecture, design, process |
| `ARCHITECTURE.md` | Overwrite current state | System structure, modules, or data flow meaningfully changed |
| `TECH_DEBT.md` | Append-only | A shortcut was knowingly taken or a future fix was deferred |
| `ONBOARDING.md` | Overwrite current state | Setup process, environment requirements, or first-run steps changed |

### Update format for append-only files:

```markdown
## [YYYY-MM-DD] — Short title
**What:** What was decided or deferred.
**Why:** The reason behind it.
**Trade-off:** What was given up or what risk is accepted.
```

### When NOT to update:
- Routine changes (fixing a typo, updating a variable name, adding a test)
- Changes that are fully self-evident from the code
- Changes that reverse a previous decision (update the original entry instead of appending a new one)

## Communication Rules

### 11. Goal-Driven Execution
Every task should have a stated success criterion before work begins. "Make it work" is not a success criterion. Confirm with the user what "done" looks like before starting.

### 12. No Silent Drift
If the codebase, requirements, or context has drifted from what `CLAUDE.md` or `ARCHITECTURE.md` describes, flag it immediately. Do not silently adapt to the drift — surface it so the tracking files can be updated.

---

## What This File Does Not Cover

- Project-specific tech stack, commands, or conventions → `CLAUDE.md`
- What the project is and what problem it solves → `PROJECT_BRIEF.md`
- History of architectural decisions → `DECISIONS.md`
- Known technical debt → `TECH_DEBT.md`
- How to set up and run the project → `ONBOARDING.md`
